# Агент адаптации резюме — документация

## 1. Что это

Инструмент для аутстафф/аутсорс компании (Outkod). Берёт резюме специалиста и адаптирует его под конкретный запрос заказчика — так чтобы нужный опыт читался явно, но резюме выглядело как написанное самим специалистом, а не подогнанное под вакансию.

---

## 2. Стек

- **Python** — основной язык
- **Streamlit** — веб-интерфейс
- **Gemini 2.5-flash** — LLM (через OpenAI-совместимый API `generativelanguage.googleapis.com/v1beta/openai/`)
- **python-docx** — генерация .docx на выходе
- **Supabase** (облако) или локальные JSON-файлы — хранение специалистов

---

## 3. Архитектура — файлы

| Файл | Назначение |
|------|------------|
| `agent.py` | Вся логика и промпты (три сценария), работа с Gemini |
| `app.py` | Streamlit UI — загрузка резюме, ввод брифа, скачивание результата |
| `bot.py` | Telegram-бот |
| `database.py` | CRUD специалистов: Supabase или локальные JSON в `specialists/` |
| `docx_utils.py` | Генерация выходного .docx, парсинг PDF/DOCX на входе |

---

## 4. Три сценария агента

### Scenario A — Convert (`convert_to_corporate`)
PDF или DOCX резюме → структурированный корпоративный JSON.

Когда резюме скудное (нет задач, достижений, описаний) — агент дописывает их из контекста. Когда резюме богатое — только структурирует.

Промпты: `CONVERT_SYSTEM`, `CONVERT_PROMPT`

### Scenario B — Tailor (`tailor_resume`)
Берёт сохранённое JSON-резюме специалиста + бриф заказчика → возвращает адаптированное резюме.

Агент меняет только то что нужно: перестраивает summary, переставляет навыки, добавляет задачи и технологии из брифа в совместимые проекты. Не трогает даты, названия компаний, команду.

Промпты: `TAILOR_SYSTEM`, `TAILOR_PROMPT`

### Scenario C — Humanize (`humanize_resume`)
Убирает роботизированность после адаптации: варьирует стартовые глаголы, убирает нагромождения терминов, делает текст живым. Факты и цифры не трогает.

Промпты: `HUMANIZE_SYSTEM`, `HUMANIZE_PROMPT`

---

## 5. Ключевые правила адаптации

### Что запрещено (абсолютно)
- Менять или придумывать даты
- Менять или придумывать названия компаний
- Объединять несколько мест работы в один проект
- Копировать фразы из брифа дословно
- Добавлять несовместимые технологии (PHP в .NET-проект и т.д.)
- Смягчать прямое владение до "взаимодействовал с командой X для настройки Y" — если оригинал показывает что специалист делал это сам

### Что обязательно
- Summary — только третье лицо, без "я/мне/специализируюсь"
- Если технология из брифа совместима с проектом — добавить и в навыки, и в задачи (не только в стек)
- Глагольный род (мужской/женский) определяется по имени и выдерживается везде

---

## 6. Правила дополнения стека

Агент добавляет технологии из брифа в проекты где они **технически совместимы** — даже если их не было в оригинале.

### .NET-стек
- **GraphQL** — совместим с любым ASP.NET Core проектом где есть REST API. Добавлять как задачу (Hot Chocolate, graphql-dotnet, DataLoader, mutations/subscriptions), не просто в стек.
- **Entity Framework Core** — совместим с любым .NET проектом где есть БД. Наличие в стеке **не считается** покрытием — нужна задача с конкретикой: Fluent API, миграции, DbContext, N+1, eager/lazy loading.
- Serilog, Keycloak, MinIO, MassTransit, Polly, MediatR, Bitbucket — совместимы с соответствующими .NET/микросервисными проектами.

### DevOps-стек
- Terraform, Helm, ArgoCD, Ansible, Vault, SonarQube — совместимы с Kubernetes/CI/CD проектами.

### Общее правило
Если технология добавлена в стек — **обязательно** добавить задачу что специалист с ней делал. Нельзя остановиться на упоминании в списке стека.

---

## 7. Самопроверка агента (Шаг 3 в промпте)

После адаптации агент проходит чеклист по каждой технологии из брифа:
- **Шаг A**: есть ли в навыках? Если нет и совместима — добавить.
- **Шаг B**: есть ли в задачах хотя бы одного проекта? Найти совместимый проект и добавить конкретную задачу.
- **Шаг C**: проверка глубины — нет ли "взаимодействовал" там где оригинал показывает прямое владение, есть ли технические детали уровня Senior.

## 8. Исправленные баги

### GraphQL не добавлялся в задачи
**Проблема:** Агент знал что GraphQL есть в брифе, добавлял его в навыки и стек, но не писал задачу что специалист с ним делал. Наличие в стеке считалось достаточным.

**Исправление:** GraphQL добавлен в правила совместимости .NET-стека (`TAILOR_SYSTEM`, раздел ПРАВИЛА ДОПОЛНЕНИЯ СТЕКА). Явно прописано что нужна задача с конкретным сценарием (Hot Chocolate, схемы, мутации, подписки, DataLoader).

### EF Core не добавлялся в задачи
**Проблема:** EF Core присутствовал в стеке проектов из оригинала, и агент считал это покрытием требования из брифа. Конкретной задачи про работу с ORM (Fluent API, миграции, оптимизация запросов) не добавлялось.

**Исправление:** EF Core добавлен в правила совместимости с явным указанием: наличие в стеке **не считается** покрытием. Добавлены примеры задач.

---

## 9. Известные нерешённые проблемы

### Angular — нет предупреждения об отсутствии
Если бриф требует Angular, а у специалиста его нет — агент корректно не добавляет Angular (нельзя выдумывать опыт). Но в `match_notes` никак не сигнализирует об этом критическом несоответствии. Заказчик видит резюме без Angular и не понимает почему.

### Проект может остаться полностью без изменений
Если агент не нашёл что добавить в конкретный проект (нет совместимых технологий из брифа) — он оставляет его нетронутым. В некоторых случаях мог бы добавить архитектурный контекст (Clean Architecture, паттерны) в описание, но не делает этого.
